HANDOFF — 回憶命中率：記憶層的真指標

View as Markdown View this page as plain textOpen in ChatGPT Ask ChatGPT about this pageOpen in Claude Ask Claude about this page

2026-07-18 一場實戰對話蒸出來的洞察，留給 clikae 的記憶層設計。起因：KITT 明明把「部署鑰匙會 403」的答案存進了清冊（別的 session 前一天寫的），隔天自己差點照舊配方原地重撞，被維護者一句「要不要看一下鑰匙清冊」人肉攔截。

三層模型

agent 的「回憶」不是一件事，是三層，每層獨立成敗：

  1. 存了沒（storage）——事實有沒有寫進耐久處。
  2. 找得到嗎（retrieval path）——從索引出發幾跳到達、成本多高。
  3. 想到要去找嗎（retrieval trigger）——動手前，routine 有沒有路過那個書架。

實戰失敗幾乎都發生在第 3 層。第 1、2 層的失敗看得見（檔案不存在、索引斷鏈），第 3 層的失敗從裡面看不見：錯的記憶跟對的記憶手感一模一樣，「自我感覺記得」是最危險的狀態。存得再好，沒觸發查閱就等於沒存。

兩種補丁（各治一層）

  • 治第 2 層：快答格。把高頻答案壓進索引頂端（「⚡ 最快答案」區），查閱成本趨近零——成本夠低，順手查的機率才會高。
  • 治第 3 層：SOP 化觸發。把「動 X 前先查 Y」寫成程序，把回憶從「靠自覺」變「照流程」。例：「動鑰匙前先看清冊快答格」「寫新記憶前先搜舊的」「動 Stripe 前必讀紅線」。觸發詞綁行為類別（動鑰匙／動定價／動部署），不綁具體事實。

對 clikae 的含義

  • 記憶層產品的關鍵指標不是存量，是「動手前的查閱命中率」：agent 在做出會被既有記憶改寫的決定之前，有多少比例真的查了？存量大而命中率低＝倉庫滿但沒人去，體感就是「它明明記過卻還是犯」。
  • 維護者在這次事件裡扮演的角色＝人肉第 3 層觸發器（「要不要看清冊？」）。產品目標是讓這個角色消失：結構要能在正確時機把記憶推到 agent 面前，而不是等 agent 想起來去拉。
  • 已有的對應設計可以往這個框架收：MEMORY.md 常駐索引＝L1、hub 按需讀＝L2、單檔事實＝L3；紅線「hub 沒載也要守」＝把最高風險事實直接釘死在 L1，等於承認第 3 層不可靠時的保險。
  • 可能的量測法（未驗證）：抽樣 session transcript，找「做了 X 但沒先讀 Y」的事件（X、Y 從 SOP 對照表來），命中率＝有查的 ÷ 該查的。冷讀稽核（[cold-read-memory-audit] 的精神）可以覆用在這。
