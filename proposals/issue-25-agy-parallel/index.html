Issue #25 — agy 同帳號平行調度（設計稿 / design note）

View as Markdown View this page as plain textOpen in ChatGPT Ask ChatGPT about this pageOpen in Claude Ask Claude about this page

狀態：設計討論，未拍板。結論傾向「不做多帳號平行（已證偽）；同帳號多 worker 條件式可做，但價值有限、優先級低」。 範圍：只談 agy（Antigravity CLI）。claude / codex 的平行另案。 相關碼：lib/commands/antigravity.sh、lib/targets/antigravity.sh、docs/dogfood-agy-headless.md、HANDOFF.md §「Dry-tank DURING a parallel multi-engine burn」。

1. 問題重述

Issue #25 想問的是：能不能讓 clikae 平行調度多個 agy worker，把可平行的大任務（例如整本字典翻譯、批次摘要）一次 fan-out 出去、加速完成，同時延續 clikae 的「跨油箱燒、不吃主線預算」核心價值。

這裡其實藏了兩個不同的命題，必須先拆開，否則會把一條死路跟一條活路混在一起討論：

  • 命題 A：多帳號平行 — 同時跑多個 agy worker，各自登入不同 Google 帳號，分別燒各自的訂閱桶。直覺上這最香（吞吐 ×N、額度 ×N）。
  • 命題 B：同帳號多 worker — 同時跑多個 agy worker，全部用同一個帳號，靠 $HOME 隔離各自的狀態（session / 工作目錄 / log），但共燒同一個訂閱桶。

本稿的核心結論：A 在 agy 上技術不可行（已實測證偽，別再追）；B 技術上勉強可行，但配額共享讓「平行」的好處大打折扣，屬於條件式、低優先級。

2. 為何「多帳號平行」（命題 A）不可行 —— 技術根因

這不是猜測，是這台機器上實測 + 讀碼確認的結論（見 agy-gemini-shared-quota 記憶、lib/commands/antigravity.sh 註解、lib/targets/antigravity.sh 抬頭）。

2.1 根因一句話：agy 的登入是 machine-wide 全域單一，不跟 $HOME 走

clikae 切換帳號的標準模型是「每個 shell 指一個 per-profile config dir」（CLAUDE_CONFIG_DIR、CODEX_HOME…）。這要求 CLI 的身分能被某個 env var 或 flag 重新導向。agy 不符合：

  • agy 的 狀態目錄 確實跟著 $HOME（os.UserHomeDir → ~/.gemini），所以搬 $HOME 能隔離 session / log / 工作狀態。
  • 但 agy 的 登入（Google OAuth） 存在一個 machine-wide 的 macOS Keychain item（service gemini、account antigravity），不在 ~/.gemini 裡。沒有任何 env（ANTIGRAVITY_EXECUTABLE_DATA_DIR、GEMINI_HOME…）或 flag 能把這個登入重新導向。

實測定案（務必納入，曾一度誤判可並行後收回）：「搬 $HOME 只搬『狀態』、不搬『auth』；auth 是全域單一，只認最後登入的帳號。」 所以即使你開三個 $HOME 不同的 shell，三個 agy 仍然全部讀同一個 Keychain 登入 = 同一個帳號。

2.2 clikae 現在怎麼處理「多帳號」—— 用全域排他換、不是平行

lib/commands/antigravity.sh 的多帳號模式（opt-in，takeover ~/.gemini）面對這個限制，採取的是全域、一次一個的解法：

  • 切 tank = 把 ~/.gemini symlink 重指到該 tank 的 slot dir，並把該 tank 的登入從 clikae 命名空間的 Keychain slot 複製回那個 canonical item（_agy_kc_stash / _agy_kc_restore，Keychain↔Keychain，token 不落地）。
  • 因為 canonical Keychain item 只有一個，這個切換是 GLOBAL：一次只有一個 agy tank 活著、跨所有終端機。碼裡甚至有 _agy_assert_not_running 守衛：agy 在跑時拒絕換 symlink，因為會把活的 session 弄壞。

換句話說，clikae 官方切換才是真換帳號，而且本質上是 machine-wide 排他。這正是「多帳號平行」的反面：要平行就得同時有多個 canonical 登入，但 agy 只給你一個槽。

2.3 結論：命題 A 是死路

  • 多開幾個 $HOME 不同的 agy → 全部同一帳號（auth 全域單一）。
  • 要不同帳號 → 必須輪流重寫那唯一的 Keychain 登入 → 本質序列、不可並行。
  • 所以 agy 的「多帳號平行」在現有 agy 設計下無法達成。 這條別再追；要追只能去等 Google 官方給 per-profile auth（不在 clikae 控制範圍）。

對照：claude / codex 可以多帳號平行（各 shell 各自 CLAUDE_CONFIG_DIR / CODEX_HOME，HANDOFF.md 的「全油箱同步燒」就是 claude+codex 多 tank 真平行）。agy 是唯一被卡在這條限制上的引擎。

3. 「同帳號多 worker」（命題 B）可行性分析

既然多帳號平行死了，剩下唯一能談的平行就是 B：同一個帳號、多個 worker、靠 $HOME 隔離狀態、共燒同一桶。

3.1 技術上可不可行？—— 勉強可行，但要繞開 clikae 現有守衛

  • 狀態隔離 OK： $HOME override 能給每個 worker 各自的 ~/.gemini（session、工作目錄、log 各自獨立）。一個 worker 跑題、卡住、被砍，不影響另一個。技術上站得住。
  • 共用登入 OK 但脆弱： 多個 worker 讀同一個 canonical Keychain 登入。讀 token 本身可並行（只讀不寫），所以理論上 N 個 agy 可以同時拿同一帳號跑。
  • ⚠️ 但會撞到 clikae 現有的全域模型： clikae 的 multi-account 模式把 ~/.gemini 當成單一 symlink在切，而且 _agy_assert_not_running 明確拒絕在 agy 跑的時候動 symlink。要做 B，不能走現有的 symlink-swap 機制，得另闢一條「per-worker HOME=... 直接起 agy、完全不碰 ~/.gemini symlink」的路徑。也就是 B 跟現有多帳號模式是兩套不相干的東西，不能混用。
  • 登入前提： worker 用的那個帳號，必須是當前 canonical Keychain item 裡那個（最後登入的）。你不能在 B 模式裡同時用兩個不同帳號（那又繞回死掉的命題 A）。所以 B 的前置是「先用 clikae 切到想燒的那個 tank、確認登入是它，然後才 fan-out N 個同帳號 worker」。

3.2 配額共享的後果 —— 這是 B 價值的致命傷

B 的 N 個 worker 共燒同一個訂閱桶。後果：

  • 沒有額度加成。 平行多帳號的香味（額度 ×N）在 B 完全不存在。你只是把同一桶油更快地燒掉。
  • 更快撞限額。 agy 每個 request 的 dry 速度本來就快（dry-run 實測 23–25k token/request，HANDOFF + agy-gemini-shared-quota 記憶）。N 個 worker 並行 = 桶被以 ×N 速度抽乾，更早吃到 RESOURCE_EXHAUSTED (429) Individual quota reached（agy 的 dry 訊號只在 ~/.gemini/antigravity-cli/cli.log 的 E-level line，見 lib/targets/antigravity.sh）。
  • 限額是「Individual quota」= 跟著帳號，不是跟著 process。 所以平行不會給你更多總量，只會把總量更快用完，外加多個 worker 同時撞牆、同時掉任務的協調成本。

一句話：B 拿「吞吐（wall-clock 更快）」換「更早 dry + 協調複雜度」，而且完全拿不到「更多總額度」。

3.3 那 B 到底有沒有價值？

有，但很窄，而且大多能用更簡單的東西取代：

  • 唯一真實的好處 = wall-clock 縮短：若任務天生可切片（N 塊互不相依），且桶夠大、N 塊加起來燒不爆，那 N 個 worker 並行確實比序列快。docs/dogfood-agy-headless.md 裡 agy 自己 spawn 5 個 subagent 平行翻譯，就是這個形狀——但注意那是 agy 行程內部自己開 subagent，不需要 clikae 介入多 $HOME。
  • 反例（B 的價值被吃掉的情況）：
    • 任務小 → 序列就夠，不值得 fan-out 的工程。
    • 任務大到會燒爆桶 → 平行只是更快撞限額，沒幫上忙，反而要處理「半路 dry、掉一半任務」（HANDOFF 那次 burn 就是一個 tank 半路 dry、靠手動 relay 補）。
    • 想要的是「不吃主線預算」→ 這跟平不平行無關，序列 offload 就已經達成（dogfood 已證）。

4. 建議

4.1 多帳號平行（命題 A）：不做。標記為已知不可行。

  • 在 agy 文件（docs/）與 issue #25 明寫：agy 無法多帳號平行，技術根因是「auth 全域單一、不跟 $HOME」。避免未來又有人（或又有一個 AI）重新發明這條死路。
  • clikae tanks / board 上的 agy row 應標示為 non-parallel / global-single（HANDOFF §「Grunt-dispatch friction」第 3 點已抱怨過 env agy 與 tanks 列 agy 的不一致——順手一起釐清）。

4.2 同帳號多 worker（命題 B）：預設不做；只在明確條件下才考慮，且優先級低。

不建議現在實作。理由：價值窄（只省 wall-clock）、拿不到額度加成、要新開一條繞過現有 symlink 模型的路徑、還要處理「共桶半路 dry」的協調——投報比差。

若日後真要做，前置條件全部滿足才動：

  1. 有反覆出現的真實任務形狀 = 「大、可乾淨切片、桶燒得起 N 塊」。沒有這個就是過度工程。
  2. 任務必須冪等 + 以 artifact 驗收（HANDOFF 結論：掉的那塊能原地重發）。
  3. 必須有 agy dry 偵測接上（cli.log 的 429 line），讓某個 worker 撞限額時能停掉整批、不要 N 個一起空轉燒（agy -p 撞限額會 exit 0 + 空輸出，不能信 exit code）。
  4. 設計上把它做成獨立於現有 multi-account symlink 模式的東西，不要污染 _agy_* 那套全域切換邏輯。

4.3 更該優先的替代方案（比 B 划算）

  • agy 行程內 subagent：要平行翻譯/批次，直接讓單一 agy agentic 地自己 spawn subagent（dogfood 已證 5 個並行），clikae 完全不必管多 $HOME。把力氣花在「寫好 headless 配方」（死任務 + 封探索 + 拉長 --print-timeout）比花在多 worker 編排划算。
  • 跨引擎平行才是真平行：要 fan-out + 額度加成，把可平行的塊分給 claude / codex 的多 tank（那些是真能多帳號平行的），agy 在裡面當其中一個序列 worker 就好。這正是 HANDOFF「全油箱同步燒」的形狀——agy 在那場是 global single，剛好對。

5. 若真要做 B —— 實作草圖

僅供未來參考；不建議現在落地。

5.1 形狀

新增一個獨立子命令（不要塞進 cmd_antigravity 的 symlink 流程），概念如 clikae agy --fan-out N -- <task-template>，或交給 burn/pool 層處理。核心機制：

# 概念草圖（非可執行碼）：N 個同帳號 worker，各自獨立 $HOME，共燒同一桶。
# 前置：使用者已用 `clikae agy <tank>` 把 canonical 登入切到目標帳號。

_agy_fanout() {
  local n="$1"; shift            # worker 數
  local task_dir="$1"; shift     # 放每個 worker 的 in/out artifact（建議 /tmp，避開 iCloud 延遲）

  # ⚠️ 不碰 ~/.gemini symlink、不呼叫 _agy_kc_stash/_agy_kc_restore。
  # 每個 worker 一個臨時 HOME，只隔離狀態；auth 仍讀全域 canonical Keychain item（同一帳號）。
  for i in $(seq 1 "$n"); do
    local whome; whome="$(mktemp -d)"
    # 把 canonical 登入「種」進這個 worker 看得到的地方？
    #   macOS：登入在 Keychain（machine-wide），worker 直接讀得到 → 不必複製。
    #   Linux：登入在 ~/.gemini 檔案 → 需把當前 active 的 ~/.gemini 內容 cp 進 $whome/.gemini。
    (
      HOME="$whome" \
      agy -i "$(render_task "$i")" \
          --dangerously-skip-permissions --add-dir "$task_dir" \
          --print-timeout 900s \
        > "$task_dir/worker-$i.out" 2>&1 &
    )
  done
  wait

  # 收尾：逐 worker 驗 artifact；任一缺檔 → 掃它的 out / 全域 cli.log 看是不是撞 429。
  # 撞限額（exit 0 但空輸出 / cli.log 有 RESOURCE_EXHAUSTED）→ 整批標 dry、把掉的塊回報給 orchestrator 重發。
}

5.2 關鍵實作注意

  • macOS vs Linux 的 auth 行為不同：macOS 登入在全域 Keychain，N worker 不必複製、天然共享；Linux 登入在 ~/.gemini 檔案，每個 $HOME worker 要先 cp 一份當前登入進去（且這份是唯讀地共用同一帳號——不要讓 worker 各自重登，否則 race）。
  • 絕不重用 _agy_assert_not_running 的反面：現有守衛是「agy 在跑就拒絕動 symlink」，B 刻意不動 symlink，所以不衝突——但也因此 B 不能跟使用者正在跑的「正常 agy 多帳號 session」並存（會搶同一個 canonical 登入的當前帳號）。文件要講清楚：fan-out 期間視同獨佔該帳號。
  • dry 偵測：每個 worker 的 ~/.gemini/antigravity-cli/cli.log（注意各 worker 自己的 $HOME 下各有一份）要 tail -F（agy 每 run 重指 log 檔，see target_limit_log_path 註解）。
  • artifact-first：所有 worker 寫檔而非靠 stdout 收（dogfood 教訓：agy -p 大輸出會「活著但吊住」）。in/out 一律放 /tmp，避開 iCloud 讀延遲（HANDOFF 的 codex 卡死同源）。
  • 封探索 + 長 timeout：每個 worker 的 prompt 都要「死任務 + 硬邊界 + 別查 git」，否則 agy 的 agentic 自主性會自己跑題把 --print-timeout 燒乾（dogfood 第 2 點）。

6. 一句話總結

agy 多帳號平行 = 死路（auth 全域單一），別追；agy 同帳號多 worker = 技術勉強可行但只省時間、不加額度、要繞過現有模型，投報比差，預設不做。要平行就走 agy 行程內 subagent 或把可平行塊丟給能真平行的 claude/codex 多 tank。
